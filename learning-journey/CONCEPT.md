# Laravel Core Concepts & Paradigms Reference

รวบรวมทฤษฎีและแนวคิดเชิงสถาปัตยกรรม (Architecture) ของ Laravel เพื่อตอบคำถามเชิงลึกในช่วง Code Review

---

## 🔒 1. การควบคุมสิทธิ์ (Authorization): Gates vs Policies

*   **Gates (ประตู):** เหมาะสำหรับการเช็คสิทธิ์แบบ Global หรือ Action-based (ไม่ผูกกับข้อมูลชิ้นใดชิ้นหนึ่ง) เช่น "ผู้ใช้คนนี้เป็น Admin ใช่ไหม?", "ผู้ใช้คนนี้มีสิทธิ์เข้าถึงแดชบอร์ดระบบหรือไม่?" (เทียบเท่ากับ Global Guard / Role Check ใน NestJS)
*   **Policies (นโยบาย):** เหมาะสำหรับการเช็คสิทธิ์แบบ Resource-based (ผูกกับ Eloquent Model ตัวใดตัวหนึ่งโดยเฉพาะ) ซึ่งตรงกับโปรเจกต์ HomeEase ของเรามากที่สุด เช่น "ผู้ใช้คนนี้เป็นเจ้าของ Booking ใบนี้จริงหรือไม่? ถึงจะยอมให้เข้าดูหรือกดยกเลิกบริการได้" (เทียบเท่ากับ CASL / Resource Ownership Guard ใน NestJS)

---

## 🗄️ 2. กลไกการรันข้อมูลฐานข้อมูล (Laravel Migrations Mechanics)

### A. ลำดับการรันของ Migration (Execution Order)
*   Laravel ใช้ตัวระบุวันที่และเวลา (Timestamp) นำหน้าไฟล์ เช่น `YYYY_MM_DD_HHMMSS_create_table_name.php` ในการตรวจสอบความเก่าใหม่ของคำสั่ง
*   **ลำดับประมวลผล:** เรียงจาก Timestamp ที่เก่าที่สุดไปยังใหม่ที่สุด หาก Timestamp เท่ากัน (เช่น สร้างขึ้นในวินาทีเดียวกันจากการรันคำสั่งหลายคลาสพร้อมกัน) Laravel จะสั่งรันเรียงตามตัวอักษรภาษาอังกฤษ (Alphabetical Order)
*   **ข้อผิดพลาดทั่วไป (Common Pitfall):** การสร้างตารางที่มีความสัมพันธ์แบบเชื่อมโยงคีย์นอก (Foreign Keys) หากตารางปลายทางเกิดถูกตั้งชื่อให้อยู่ท้ายสุดตามตัวอักษร (เช่น `services`) แต่ตารางที่ต้องเรียกใช้งานถูกสร้างขึ้นก่อน (เช่น `bookings`) จะทำให้ DDL พังทันที

### B. ความทนทานและการทำงานเข้ากันได้กับฐานข้อมูลอื่น (Database Portability)
*   ฐานข้อมูลบางประเภท เช่น **SQLite** จะมองข้ามการเปิดระบบเช็ค Foreign Key เป็นการชั่วคราวขณะรันสคีมา ทำให้รันผ่านได้แม้ลำดับการรันจะผิดพลาด
*   แต่ระบบฐานข้อมูลประเภท Enterprise เช่น **MySQL, MariaDB, หรือ PostgreSQL** มีการตรวจสอบ Referential integrity อย่างเคร่งครัดตั้งแต่ขั้นตอนคอมไพล์สกีมา การจัดลำดับไฟล์ Migration จึงสำคัญอย่างยิ่งเพื่อให้ระบบรองรับการย้ายไปยังฐานข้อมูลประเภทอื่น (Cross-Database Compatibility)

### C. การป้องกันข้อมูลในความสัมพันธ์ (Referential Action Guards)
*   `onDelete('cascade')`: เมื่อลบแถวข้อมูลในตารางแม่ แถวข้อมูลในตารางย่อยที่มีความสัมพันธ์อยู่จะถูกลบตามทันที (เช่น ลบหมวดหมู่ ให้ลบบริการและทักษะช่างออกให้หมด)
*   `onDelete('restrict')`: ห้ามไม่ให้ลบแถวข้อมูลในตารางแม่ หากยังมีข้อมูลอ้างอิงเหลืออยู่ในตารางย่อย (เช่น ห้ามลบข้อมูลช่างหรือบริการ หากช่าง/บริการนั้นเคยถูกนำไปกดจองในใบจอง `bookings` เพื่อรักษาประวัติการทำธุรกรรมเดิมไว้)

## 🛡️ 3. ระบบ Mass Assignment & ความปลอดภัยระดับ ORM

### A. ความหมายของ Mass Assignment
*   คือการที่เราส่งข้อมูลแบบเป็นชุดอาร์เรย์ (เช่น `$request->all()`) เข้าไปบันทึกในโมเดลตรง ๆ ผ่าน `Model::create($array)` หรือ `$model->fill($array)`
*   **Whitelist (`$fillable` / `#[Fillable]`):** ระบุคอลัมน์ที่อนุญาตให้บันทึกหรือแก้ไขด้วยอาร์เรย์ได้
*   **Blacklist (`$guarded` / `#[Guarded]`):** ระบุคอลัมน์ที่ห้ามแก้ไขด้วยอาร์เรย์ (คอลัมน์อื่น ๆ ที่ไม่อยู่ในรายชื่อจะบันทึกได้ทั้งหมด)

### B. เปรียบเทียบสถาปัตยกรรมความปลอดภัย: NestJS vs Laravel
*   **NestJS (ฝั่ง ORM):** Prisma/TypeORM ไม่มีระบบจำกัดการเขียนข้อมูลระดับคลาสโมเดล เพราะความปลอดภัยถูกกำหนดไว้ที่ส่วนรับข้อมูลด้านบนสุด (Controller) ผ่าน Class-Validator & DTO
*   **Laravel (ฝั่ง ORM):** ความปลอดภัยจะทำงานตั้งแต่ระดับฐานข้อมูล (Eloquent) เพื่อทำหน้าที่เป็นด่านป้องกันสุดท้าย (Last Line of Defense) ป้องกันไม่ให้แฮกเกอร์เขียนทับคอลัมน์สำคัญ (เช่น `is_admin`, `balance`) จากการ Overposting

## 🏭 4. การจัดการข้อมูลจำลอง (Factories, Seeders & PHP Traits)

### A. ความแตกต่างระหว่าง Seeders และ Factories
*   **Seeders:** ออกแบบมาเพื่อลงข้อมูลประเภท Master Data หรือ Lookup Data ที่เป็นข้อมูลตายตัวในระบบ (เช่น สิทธิ์ Roles, หมวดหมู่บริการ Categories)
*   **Factories:** ออกแบบมาเพื่อจำลองข้อมูลแบบ Dynamic/Random จำนวนมากเพื่อการทดสอบ (เช่น สุ่มสร้างข้อมูลช่าง 1,000 คน พร้อมเบอร์โทรและรีวิวจำลอง) โดยทำงานร่วมกับไลบรารี **Faker** ในตัว

### B. กลไกเบื้องหลัง: PHP Traits กับ `HasFactory`
*   **Trait ใน PHP:** คล้ายกับการทำ Multiple Inheritance (หรือ Mixin ในภาษาอื่น) เพื่อให้คลาสใด ๆ สามารถหยิบเอาเมธอดสำเร็จรูปใน Trait ไปแปะใช้งานได้โดยไม่ต้องสืบทอด (Extend) จากคลาสแม่คนละสาย
*   **`HasFactory` Trait:** เมื่อคลาสโมเดลเรียกใช้ `use HasFactory;` ตัว Trait นี้จะสแกนหาชื่อ Factory ที่มีรูปแบบการตั้งชื่อสอดคล้องกัน (เช่น `App\Models\Provider` จะจับคู่กับ `Database\Factories\ProviderFactory`) และทำการ Inject เมธอด `factory()` เข้ามาในคลาสโมเดลให้โดยอัตโนมัติ 
*   **ความเชื่อมโยงกับ NestJS:** ใน NestJS หากเราไม่มีระบบ Seeder ในตัวที่ทรงพลังแบบนี้ เรามักเขียนโมดูลสุ่มเองผ่าน Class / Service และฟังก์ชันจำลองข้อมูลที่ต้องประดิษฐ์ขึ้นใหม่เองในแต่ละโปรเจกต์

## 🔑 5. API Authentication Mechanics: `Auth::attempt()` vs `Hash::check()`

ในการตรวจสอบความถูกต้องของข้อมูลก่อนออกโทเค็น Laravel มีเครื่องมือในการ Verify คีย์หลักและรหัสผ่านสองระดับที่มีขอบเขตหน้าที่ (Scope of Responsibility) ต่างกัน:

### A. เปรียบเทียบขอบเขตการทำงาน (Functional Scope Comparison)

| มิติการเปรียบเทียบ | `Hash::check($plain, $hashed)` | `Auth::attempt($credentials)` |
| :--- | :--- | :--- |
| **ระดับเลเยอร์ (Layer)** | Low-level Utility (เครื่องมือช่วยตรวจสอบแฮช) | High-level Authentication Guard (ระบบตรวจสอบสิทธิ์) |
| **การทำงานฐานข้อมูล** | **ไม่มี** (ตรวจสอบข้อมูลเชิงคณิตศาสตร์ไร้รัฐเท่านั้น) | **มี** (เขียนคำสั่ง SQL Query เพื่อค้นหา User อัตโนมัติ) |
| **การเปรียบเทียบเทียบเท่า NestJS** | คล้ายกับคำสั่ง `bcrypt.compare()` หรือ `argon2.verify()` | คล้ายการทำงานของ **Passport Local Strategy** |
| **สิ่งที่ต้องการในการทำงาน** | ต้องการรหัสผ่านธรรมดาและค่าแฮชเดิมที่ดึงมารอไว้แล้ว | ต้องการอาร์เรย์ของข้อมูล เช่น `['email' => ..., 'password' => ...]` |
| **ผลข้างเคียง (Side Effects)** | คืนค่า `true` / `false` เท่านั้น ไร้รัฐโดนสิ้นเชิง | หากล็อกอินผ่านเว็บ จะเขียนไฟล์ Session และคุกกี้ให้อัตโนมัติ |

### B. กลไกการทำงานของ Laravel Sanctum (Token-Based Auth)
*   **การเก็บรักษาโทเค็น:** Sanctum จะทำการเก็บโทเค็นของระบบไว้ในตาราง `personal_access_tokens` โดยมีการเข้ารหัสด้วย SHA-256 ก่อนเก็บลงฐานข้อมูล เพื่อป้องกันไม่ให้ข้อมูลโทเค็นรั่วไหลหากฐานข้อมูลโดนแฮก
*   **กระบวนการตรวจสอบ (Authentication Flow):**
    1.  เมื่อ Client แนบ Header `Authorization: Bearer <token>` เข้ามาในระบบ
    2.  `auth:sanctum` Middleware จะถอดโทเค็นออกมาและนำไปแฮชเทียบกับตาราง `personal_access_tokens`
    3.  หากจับคู่สำเร็จ จะทำการดึงข้อมูลโมเดล `User` เจ้าของโทเค็นขึ้นมารอไว้ในหน่วยความจำ
    4.  ฝั่งนักพัฒนาสามารถเรียกใช้อ็อบเจกต์ผู้ใช้คนนั้นได้ทันทีใน Controller ผ่าน `$request->user()`

### C. รายละเอียดโครงสร้างตาราง `personal_access_tokens`

นี่คือคอลัมน์ทั้งหมดของตารางเก็บโทเค็นของ Sanctum ที่ถูกสร้างขึ้นมาในระบบ:

| ชื่อคอลัมน์ (Column) | ประเภทข้อมูล (Data Type) | หน้าที่ในการทำงาน |
| :--- | :--- | :--- |
| **`id`** | BigInt (PK) | ไอดีลำดับหลักของเรคคอร์ดโทเค็น |
| **`tokenable_type`** | String (varchar) | คลาสโมเดลที่ผูกกับโทเค็นนี้ เช่น `App\Models\User` (รองรับระบบ Polymorphic) |
| **`tokenable_id`** | BigInt | ไอดีของเรคคอร์ดโมเดลนั้น เช่น `1` (เมื่อประมวลผลคู่กับคอลัมน์ด้านบน จะได้เจ้าของโทเค็น) |
| **`name`** | String (varchar) | ชื่อที่ใช้เรียกกลุ่มโทเค็นนี้ เช่น `auth_token` |
| **`token`** | String (varchar, Unique) | ค่าโทเค็นตัวจริงที่ถูกแปลงให้เป็นรหัสแฮชแบบ **SHA-256 (ความยาว 64 หลัก)** ก่อนบันทึก |
| **`abilities`** | Text (JSON) | ค่าสิทธิ์การทำธุรกรรมจำแนกตาม Token (เช่น `["*"]` เข้าถึงได้ทุกอย่าง) |
| **`last_used_at`** | Timestamp | อัปเดตเวลาล่าสุดที่ Client ส่งโทเค็นนี้เข้ามาขอใช้บริการเพื่อเช็คความเคลื่อนไหว |
| **`expires_at`** | Timestamp | ระบุวันและเวลาหมดอายุของโทเค็น หากพ้นเวลานี้แล้วโทเค็นจะถูกยกเลิกการใช้งานทันที |
| **`created_at` / `updated_at`** | Timestamp | วันเวลาการลงทะเบียนและอัปเดตแถวข้อมูลระบบ |

### D. กลไกการกำหนดเวลาหมดอายุของ Token (Token Expiration)
*   **Default Behavior:** ในค่าเริ่มต้นของ Sanctum ค่าคอลัมน์ `expires_at` จะถูกบันทึกเป็น `NULL` เสมอ ซึ่งหมายความว่า **โทเค็นจะไม่มีวันหมดอายุ** (จนกว่าผู้ใช้จะส่งคำขอ Logout เพื่อมาลบทิ้ง หรือลบผ่านระบบด้วยตนเอง)
*   **การตั้งค่าจำกัดเวลา (Expiration Config):** หากเราเข้าแก้ไขไฟล์ `config/sanctum.php` และปรับปรุงค่าดีฟอลต์ในคีย์ `'expiration'`:
    ```php
    'expiration' => 60, // กำหนดให้โทเค็นหมดอายุใน 60 นาที (1 ชั่วโมง) หลังสร้าง
    ```
    เมื่อตั้งค่านี้ ทุกครั้งที่ Sanctum ออกโทเค็นใหม่ มันจะเขียนเวลาล่วงหน้าลงในช่อง `expires_at` ให้อัตโนมัติ และเมื่อมีการยิง Token นี้เข้าด่านตรวจ ระบบจะเช็คเวลาปัจจุบันเปรียบเทียบกับฟิลด์ดังกล่าว หากผ่านกำหนดแล้วจะตีกลับทันทีเป็น `401 Unauthorized`

## 🗃️ 6. Layer แปลงผลลัพธ์ข้อมูล (API Resources & Eager Loading)

ในการสร้าง REST API ของ Laravel มีกลไกการแปลงรูปข้อมูลโมเดลและการลดภาระคิวรีฐานข้อมูล (Optimizations) ที่สำคัญและเทียบเคียงกับ NestJS ดังนี้:

### A. เปรียบเทียบ Serialization Layer: NestJS vs Laravel

*   **NestJS (class-transformer):**
    *   ใช้ Decorators เช่น `@Exclude()`, `@Expose()`, `@Type()` แปะลงใน Class Entity หรือ DTO
    *   การแปลงค่าทำงานบนระดับ CPU Memory หลังดึงข้อมูลขึ้นมาแล้ว (เช่น แปลงวันเวลา หรือซ่อนฟิลด์ลับ)
*   **Laravel (API Resources):**
    *   แยก Layer การจัดรูปข้อมูลออกมาชัดเจนเป็นไฟล์คลาสทรัพยากร เช่น `JsonResource`
    *   ช่วยให้จำกัดฟิลด์ที่จะส่งออก หรือแปลงชนิดข้อมูล (เช่น แปลงคอลัมน์ `base_price` เป็นทศนิยม `float`) ได้อย่างง่ายดาย
    *   **Conditional Attributes:** สามารถเลือกซ่อนหรือโชว์ฟิลด์ตามความสัมพันธ์ที่โหลดเข้ามาได้ เช่น:
        *   `$this->whenLoaded('relation')` แสดงความสัมพันธ์ต่อเมื่อเรียกผ่าน Eager Loading เท่านั้น (ป้องกัน N+1)
        *   `$this->whenCounted('relation')` แสดงจำนวนแถวความสัมพันธ์ที่นับมาจากคิวรี (ใช้คู่กับ `withCount()`)

### B. ปัญหา N+1 Query และแนวทางการแก้ไข (Eager Loading)

เมื่อเราดึงโมเดลหลักมา $N$ แถว (เช่น ดึง Services มา 10 รายการ) แล้วพยายามวนลูปเพื่อเรียกข้อมูลหมวดหมู่ (`$service->category->name`) Laravel จะยิงคำสั่ง SQL เพิ่มอีก $N$ ครั้งเพื่อดึงตารางหมวดหมู่ย่อย ทำให้เกิดการคิวรีฐานข้อมูลซ้ำซ้อนรวมเป็น $N+1$ ครั้ง

*   **การป้องกันด้วย Eager Loading (`with()`):**
    *   เทียบเท่ากับ `relations: ['category']` ใน TypeORM / Prisma ของ NestJS
    *   สั่งให้ Eloquent คิวรีตารางแม่และตารางย่อยเชื่อมโยงกันทีเดียวด้วยคำสั่ง SQL `IN` เช่น `SELECT * FROM categories WHERE id IN (1, 2, 3)`
    *   ในโค้ดของเรา: `Service::with('category')->get()` ดึงข้อมูลทั้งหมดในคำสั่งเดียว
*   **การตรวจสอบเงื่อนไขความสัมพันธ์ด้วย `whereHas()`:**
    *   ใช้คัดกรองแถวข้อมูลของโมเดลหลัก โดยเช็คเงื่อนไขความสัมพันธ์บนตารางลูกย่อยผ่านคำสั่ง `EXISTS` ใน SQL
    *   ในโค้ดของเรา: การค้นหาช่างที่ตรงหมวดหมู่บริการจะใช้ `whereHas('categories')` เพื่อกรองผู้ให้บริการที่มีหมวดหมู่นั้น ๆ เท่านั้น
*   **การนับจำนวนความสัมพันธ์ด้วย `withCount()`:**
    *   ใช้ดึงจำนวนโมเดลลูกย่อยโดยไม่ต้องดาวน์โหลดแถวข้อมูลขึ้นมารอในหน่วยความจำ
    *   ระบบจะทำการ Select คอลัมน์ย่อยเพิ่มเข้ามาเป็น `[relation]_count` (เช่น `services_count`) ให้อัตโนมัติใน SQL Query เดียว ช่วยประหยัด Memory และ IO อย่างมาก

## 🔒 7. ระบบการจองและสิทธิ์ความปลอดภัยระดับ Resource (Booking Engine & Resource Policies)

ในการพัฒนาโมดูลการจองบริการและตรวจสอบความถูกต้อง มีหัวข้อทางเทคนิคที่สามารถเชื่อมโยงแนวคิดระหว่าง NestJS และ Laravel ดังนี้:

### A. เปรียบเทียบ Validation Layer: NestJS DTO Pipes vs Laravel Form Requests

*   **NestJS (ValidationPipe + class-validator):**
    *   สร้างคลาส DTO (Data Transfer Object) ตกแต่งฟิลด์ด้วย Decorator (เช่น `@IsString()`, `@IsDate()`) และนำมาสแกนผ่าน `ValidationPipe` ในระดับ Global หรือ Controller Guard
*   **Laravel (Form Requests):**
    *   สกัด (Extract) กฎการ Validate ออกจากคอนโทรลเลอร์ไปไว้ในคลาส Request เฉพาะ เช่น `CreateBookingRequest`
    *   **ข้อดี:** ตัวคอนโทรลเลอร์มีความ "Skinny" (ผอมบาง) มากขึ้น โฟกัสเฉพาะ Logic ธุรกิจ และควบคุมการอนุญาตสิทธิ์ระดับ Http Request ได้ในเมธอด `authorize()` ของ Form Request ทันที
    *   ตัวอย่างกฎยอดนิยม: `'appointment_datetime' => 'required|date|after:now'` การันตีเวลาจองต้องไม่ย้อนหลัง ณ เวลาปัจจุบัน

### B. สถาปัตยกรรมสิทธิ์ความปลอดภัย: NestJS Guards/CASL vs Laravel Policies

*   **NestJS (CASL / Resource Guards):**
    *   ผู้ใช้นิยมสร้าง Custom Guard (เช่น `OwnerGuard` หรือ `PoliciesGuard`) ร่วมกับ CASL เพื่อตรวจสอบว่า `user.id === resource.userId` โดยบางครั้งต้องคิวรีข้อมูลขึ้นมาเช็คระดับ Controller เสมอ
*   **Laravel (Eloquent Policies):**
    *   ผูกการประเมินสิทธิ์เข้ากับ Eloquent Model โดยตรง (เช่น `BookingPolicy` คอยเช็คโมเดล `Booking`)
    *   สามารถเรียกผ่าน Facade `Gate::authorize('action', $model)` ในคอนโทรลเลอร์
    *   **Side Effects:** หากสิทธิ์ถูกปฏิเสธ (คืนค่า `false` ใน Policy) Laravel จะทริกเกอร์ `AuthorizationException` โดยอัตโนมัติ และเซิร์ฟเวอร์จะคืนค่า `403 Forbidden` ให้ฝั่ง Client ทันทีโดยที่เราไม่ต้องเขียนโค้ด Try/Catch
    *   การนำมาประยุกต์ใช้ในโปรเจกต์:
        *   `view`: ตรวจสอบสิทธิ์เจ้าของใบจอง (`$user->id === $booking->user_id`)
        *   `cancel`: ตรวจสอบสิทธิ์ความเป็นเจ้าของ **ควบคู่กับสถานะของใบจอง** (ต้องเป็น `pending` หรือ `accepted` เท่านั้น ถึงจะยกเลิกบริการได้)

### C. การรักษาสถานะทางบัญชี (Price Snapshot Pattern)

*   **ปัญหาแบบดั้งเดิม:** หากเราบันทึกใบจองโดยวิธีอ้างอิงตรงไปยัง `services.base_price` (ผ่าน Foreign Key เท่านั้น) เมื่อวันเวลาผ่านไปและแอดมินระบบเข้ามาปรับปรุงราคาฐาน (Base Price) ของบริการนั้นๆ รายงานการเงินประวัติการจองเดิมจะคลาดเคลื่อนและไม่สอดคล้องกับค่าใช้จ่ายจริงที่ลูกค้าเคยชำระ
*   **แนวทางการแก้ไข (Price Snapshot):**
    *   ขณะสร้างบันทึกในโมเดล `Booking` เราจะคัดลอกค่าราคาปัจจุบันจาก `Service` มาเก็บบันทึกเดี่ยวๆ ในคอลัมน์ `price_charged` ของตาราง `bookings`
    *   **ประโยชน์:** ประวัติราคาถูกบันทึกแช่แข็ง (Freeze State) ณ วันเวลาที่เกิดธุรกรรม ทำให้ไม่ได้รับผลกระทบจากการปรับแต่งโครงสร้างราคาในอนาคต (Immutable Financial Audit Trail)
